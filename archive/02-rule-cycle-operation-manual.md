# ルール改善サイクル運用説明書

## 概要

本説明書は、構造化されたルール管理システム（`.cursor/rules`ディレクトリ）の日常的な運用方法について説明します。

## ルール構造の概要

```
.cursor/rules/
├── general.mdc           # 一般ルール（ルール改善サイクルの核）
├── environment.mdc       # 開発環境ルール
├── implementation.mdc    # 実装ルール
├── github.mdc           # GitHubルール
└── knowledge.mdc        # 知見の棚（ルール改善サイクルの核心）
```

## 日常的な運用フロー

### 1. 作業開始時

#### 確認事項
- [ ] 関連するルールファイルの内容を確認
- [ ] knowledge.mdcの最新の失敗例と再発防止ルールを確認
- [ ] 作業に必要なルールを宣言

#### 宣言例
```
「以下のルールを考慮しながら作業します：
- general.mdc: ルール改善サイクルの実行
- environment.mdc: 仮想環境の使用と危険操作の確認
- implementation.mdc: テストと動作確認の実施」
```

### 2. 作業中の記録

#### 問題発生時の対応
1. **即座の記録**: knowledge.mdcに失敗例を記録
2. **解決策の発見**: 解決策を見つけたら再発防止ルールを追加
3. **効率的な処理方法**: 新たな手法があれば記録

#### 記録例
```markdown
### 環境関連（追加）
- 失敗例：Pythonパッケージのバージョン競合でエラー
- 再発防止ルール：requirements.txtでのバージョン固定を必須とする
- 効率的な処理方法：pip freeze > requirements.txtの定期実行
```

### 3. 作業終了時

#### 必須作業
1. **学習内容の反映**: knowledge.mdcに今回の学びを追記
2. **ルール更新**: 必要に応じて他のルールファイルも更新
3. **遵守確認**: 考慮すべきルールを遵守できているかチェック

#### 終了宣言例
```
「作業完了。以下のルールを遵守しました：
- 仮想環境を使用してパッケージ管理
- テスト実行と動作確認を実施
- Issue管理での作業進行
新しい知見をknowledge.mdcに反映しました。」
```

## ルール更新のガイドライン

### 1. knowledge.mdcの更新

#### 更新タイミング
- 問題が発生した時
- 新しい解決策を発見した時
- 効率的な処理方法を発見した時

#### 更新形式
```markdown
### [カテゴリ]（追加日時）
- 失敗例：[具体的な問題]
- 再発防止ルール：[具体的な対策]
- 効率的な処理方法：[発見した手法]（任意）
```

### 2. 他のルールファイルの更新

#### 更新が必要な場合
- 基本方針の変更
- 新しい開発環境の追加
- 実装手法の改善
- GitHub運用の変更

#### 更新手順
1. 該当する.mdcファイルを編集
2. 変更内容をknowledge.mdcの更新履歴に記録
3. 変更の効果を次回の作業で確認

## 定期メンテナンス

### 週次メンテナンス（推奨）

#### 確認事項
- [ ] 不要になったルールの削除
- [ ] 重複するルールの統合
- [ ] 新しい知見の整理

#### 実行例
```bash
# 1. ルールファイルの確認
ls -la .cursor/rules/

# 2. 最新の更新内容確認
grep -n "追加\|更新" .cursor/rules/knowledge.mdc

# 3. ファイルサイズの確認（500行未満を維持）
wc -l .cursor/rules/*.mdc
```

### 月次メンテナンス

#### 実施内容
- [ ] ルールの効果測定
- [ ] 使用頻度の低いルールの見直し
- [ ] 新しいルールの必要性の検討

## トラブルシューティング

### よくある問題と解決策

#### 1. ルールが反映されない
**症状**: Cursorが新しいルールを認識しない
**解決策**: 
- Cursorの再起動
- `.cursor/rules/`ディレクトリの権限確認
- ファイルの文字エンコーディング確認（UTF-8）

#### 2. ルールファイルが大きくなりすぎた
**症状**: ファイルが500行を超える
**解決策**:
- 関連するルールを別ファイルに分割
- 古い情報の削除
- カテゴリ別の再整理

#### 3. 複数デバイスでの同期問題
**症状**: デバイス間でルールが異なる
**解決策**:
- Gitでのバージョン管理
- クラウドストレージでの同期
- 手動での同期確認

## 効果測定

### 定量的指標

#### 開発効率の向上
- **エラー発生率**: 同じエラーの繰り返し回数
- **問題解決時間**: 初回発生から解決までの時間
- **ルール更新頻度**: 月次のルール追加・更新回数

#### 記録例
```markdown
## 効果測定（2025年9月）
- 仮想環境エラー: 3回 → 0回（100%改善）
- テスト未実行による問題: 2回 → 0回（100%改善）
- ルール更新回数: 5回（新規知見の蓄積）
```

### 定性的指標

#### 開発体験の向上
- **ストレスの減少**: 同じ問題の繰り返しによるストレス
- **学習効果**: 新しい技術の習得速度
- **チーム協働**: 知識共有の促進

## ベストプラクティス

### 1. ルールの品質管理

#### 良いルールの特徴
- **具体的**: 曖昧な表現を避ける
- **実用的**: 実際の作業で使える内容
- **更新可能**: 継続的な改善が可能

#### 悪いルールの例
- 「なるべくテストを実行する」（曖昧）
- 「エラーを避ける」（具体的でない）
- 「良いコードを書く」（主観的）

### 2. 知識の蓄積

#### 効果的な記録方法
- **即座の記録**: 問題発生時にすぐに記録
- **具体的な記述**: 抽象的な表現を避ける
- **解決策の明確化**: なぜその解決策が有効かを記述

### 3. チームでの活用

#### 共有のポイント
- **標準化**: チーム共通のルール作成
- **教育**: 新しいメンバーへのルール説明
- **継続的改善**: チーム全体でのルール改善

## まとめ

ルール改善サイクルの運用は、以下の要素が重要です：

1. **継続的な記録**: knowledge.mdcへの定期的な更新
2. **効果的な活用**: 作業開始時のルール確認
3. **定期的なメンテナンス**: 週次・月次の見直し
4. **効果測定**: 定量的・定性的な指標での改善確認

これらの運用により、AIエージェントとの協働がより効率的になり、開発体験が大幅に向上します。
