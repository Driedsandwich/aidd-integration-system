# Cursorルール改善サイクル導入ガイド

## 概要

本ガイドは、YouTube動画「【実践】AI駆動開発を10倍快適にする【AIDD】」で示されたルール改善サイクルを、Cursorの推奨される`.cursor/rules`ディレクトリ構造に導入する方法を説明します。

## 前提条件

- Cursorの最新版を使用
- プロジェクトルートに`.cursor`ディレクトリを作成可能
- `.mdc`ファイルの編集が可能

## 推奨ディレクトリ構造

```
.cursor/
└── rules/
    ├── general.mdc           # 一般ルール（ルール改善サイクルの核）
    ├── environment.mdc       # 開発環境ルール
    ├── implementation.mdc    # 実装ルール
    ├── github.mdc           # GitHubルール
    └── knowledge.mdc        # 知見の棚（ルール改善サイクルの核心）
```

## 各ファイルの詳細設計

### 1. general.mdc（一般ルール）

```markdown
# 一般ルール

## 基本方針
- このルールシステムは「AIが同じ失敗を繰り返さない」ための学習ノートです
- 不具合や学びが出たら、knowledge.mdcに短く追記してください
- 作業終了時は必ず学習内容をknowledge.mdcに反映してください

## ルール改善サイクル
1. 指示理解 → 2. コード実行 → 3. エラー発生 → 4. フィードバック → 5. 解決策整理 → 6. ルール反映

## 作業フロー
- 作業開始時：関連するルールファイルを確認してください
- 作業中：問題が発生したら即座にknowledge.mdcに記録してください
- 作業終了時：学習内容をknowledge.mdcに反映してください

## レスポンス形式
- 作業開始時に、どのプロジェクトルールを考慮しながら作業すべきか検討し、宣言してください
- 作業終了時に、考慮すべきルールを遵守できているかチェックしてから終わってください
```

### 2. environment.mdc（開発環境ルール）

```markdown
# 開発環境ルール

## 仮想環境の使用
- Pythonプロジェクトでは必ず仮想環境を使用してください（venv / uv）
- グローバルインストールは禁止です
- 環境構築時は仮想環境の作成・有効化を最初に実行してください

## 危険操作の確認
- 削除・上書き操作は必ず確認質問をしてください
- システムファイルの変更は避けてください
- 重要なデータの削除前にバックアップを提案してください

## 環境固有の設定
- Windows環境ではPowerShellコマンドを使用してください
- パスの区切り文字は環境に応じて適切に選択してください
```

### 3. implementation.mdc（実装ルール）

```markdown
# 実装ルール

## コード変更時の必須事項
- 変更時はテストまたは動作確認手順も提示してください
- 変更内容の説明を明確に記載してください
- 影響範囲を事前に分析してください

## エラー対応
- エラー時は原因候補・再現手順・修正案をセットで示してください
- 同じエラーの繰り返しを避けるため、解決策をknowledge.mdcに記録してください
- エラーの根本原因を特定し、再発防止策を提案してください

## コード品質
- プログラム内にコメントアウトを記入することを禁止します
- 可読性を重視したコードを記述してください
- 適切なエラーハンドリングを実装してください
```

### 4. github.mdc（GitHubルール）

```markdown
# GitHubルール

## Issue管理
- 作業はIssue単位で行ってください
- 新機能実装時は必ずIssueを作成してください
- Issueのステータス管理を適切に行ってください

## Pull Request
- 1PR=1トピックの原則を守ってください
- 変更理由と範囲を明記してください
- 適切なコミットメッセージを作成してください

## ブランチ戦略
- 機能ブランチでの作業を推奨します
- メインブランチへの直接プッシュは避けてください
```

### 5. knowledge.mdc（知見の棚）

```markdown
# 知見の棚

## 失敗例と再発防止ルール

### 環境関連
- 失敗例：グローバルに依存パッケージを入れて壊した
- 再発防止ルール：仮想環境を必須とする

- 失敗例：仮想環境を無視してコマンドを実行した
- 再発防止ルール：Python実行前に仮想環境の確認を必須とする

### 実装関連
- 失敗例：テストなしでコードを変更した
- 再発防止ルール：変更前にテストを実行する

- 失敗例：危険操作を確認なしで実行した
- 再発防止ルール：危険操作前に確認を求める

### GitHub関連
- 失敗例：大きな変更を1つのPRにまとめた
- 再発防止ルール：1PR=1トピックを徹底する

## 効率的な処理方法
- よく使用するコマンドのショートカット化
- テンプレートファイルの活用
- 自動化スクリプトの作成

## 更新履歴
- 2025-09-12: 初期版作成
- [日付]: [更新内容]
```

## 導入手順

### ステップ1: ディレクトリ構造の作成

```bash
# プロジェクトルートで実行
mkdir -p .cursor/rules
```

### ステップ2: 各ファイルの作成

1. 上記の設計に基づいて各`.mdc`ファイルを作成
2. 現在の`.cursorrules`ファイルの内容を適切なファイルに移行
3. 各ファイルが500行未満になるよう調整

### ステップ3: 既存ルールの移行

現在の`.cursorrules`ファイルの内容を新しい構造に移行：

```markdown
# 移行マッピング
## General（共通） → general.mdc
## Environment（環境） → environment.mdc
## Implementation（実装） → implementation.mdc
## GitHub（運用） → github.mdc
## Knowledge（知見の棚） → knowledge.mdc
## Requirements（要件） → general.mdcに統合
```

### ステップ4: テストと検証

1. 各ルールファイルが正しく読み込まれるか確認
2. ルール改善サイクルが正常に動作するかテスト
3. 必要に応じてルールの調整

## ルール改善サイクルの実装

### 1. 作業開始時の確認

```
作業開始時に以下の確認を行ってください：
1. 関連するルールファイルの内容を確認
2. knowledge.mdcの最新の失敗例と再発防止ルールを確認
3. 作業に必要なルールを宣言
```

### 2. 作業中の記録

```
問題が発生した場合：
1. 即座にknowledge.mdcに失敗例を記録
2. 解決策を見つけたら再発防止ルールを追加
3. 効率的な処理方法があれば記録
```

### 3. 作業終了時の反映

```
作業終了時に以下を実行：
1. 学習内容をknowledge.mdcに反映
2. 新しい失敗例や解決策があれば追記
3. ルールの更新が必要な場合は適切なファイルに反映
```

## ベストプラクティス

### 1. ルールの管理

- **定期的な見直し**: 月次でルールの効果を検証
- **不要なルールの削除**: 使われなくなったルールは削除
- **ルールの統合**: 似たようなルールは統合を検討

### 2. パフォーマンスの最適化

- **ファイルサイズの管理**: 各ファイルを500行未満に保つ
- **不要な情報の削除**: 古い情報は定期的に整理
- **コンテキストの最適化**: 必要な情報のみを保持

### 3. チームでの活用

- **ルールの共有**: チームメンバーとルールを共有
- **標準化**: チーム共通のルールを作成
- **教育**: 新しいメンバーへのルール教育

## 注意点

### 1. 移行時の注意

- 既存の`.cursorrules`ファイルは削除前にバックアップ
- 移行後は動作確認を十分に行う
- 必要に応じて段階的に移行する

### 2. メンテナンス

- ルールの更新は定期的に行う
- 効果のないルールは削除する
- 新しい問題が発生したら即座にルールに反映

## まとめ

この導入により、YouTube動画で示されたルール改善サイクルを、Cursorの推奨される構造で実現できます。特に重要なのは：

1. **knowledge.mdc**による知見の蓄積
2. **段階的なルール改善**
3. **継続的な学習サイクル**

これらの要素により、AIエージェントが使用するたびに改善され、最終的に「最強」の状態に近づいていきます。
