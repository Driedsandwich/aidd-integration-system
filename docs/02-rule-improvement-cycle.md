# ルール改善サイクル実践ガイド

## 概要

本ガイドは、YouTube動画「【実践】AI駆動開発を10倍快適にする【AIDD】」で示された「ルール改善のサイクル」について、詳細な実装方法と実践的な活用方法をまとめたものです。

## 核心理念

### 基本思想
- **完璧なルールを一度で作るのではなく、使用しながら改善する仕組みを構築する**
- **使えば使うほど改善する仕組みを作る**
- **作業するたびに生まれた経験的な知見をルールに蓄積する**

### 「ルールオプス」の概念
動画のスピーカーが提唱する「ルールオプス」とは：
- ルールを最適化していくプロセス
- 使用するたびに改善される仕組み
- 最強のルールに近づけていくサイクル

## ルールの構造

### 1. ユーザールール（プロジェクト共通）
- **特徴**: プロジェクト問わず適用される
- **内容**: フルネーム、パーソナリティ等の基本設定
- **編集**: カーソルの場合、自力で編集できない可能性

### 2. プロジェクトルール（プロジェクト固有）
- **特徴**: プロジェクトごとに適用される
- **編集**: `cursor rules generate`コマンドで編集可能
- **管理**: ローカルファイルとして自由に管理可能

### 推奨するルール構造

```markdown
# Cursor Rules (project-level)

## General（共通）
- このファイルは「AIが同じ失敗を繰り返さない」ための学習ノート
- 不具合や学びが出たら、下の各章に短く追記する

## Environment（環境）
- 仮想環境を使う（例：Pythonなら `venv` / `uv`）
- 危険操作（削除・上書き）は必ず確認質問をする

## Implementation（実装）
- 変更時はテストまたは動作確認手順も提示する
- エラー時は原因候補・再現手順・修正案をセットで示す

## GitHub（運用）
- 作業はIssue単位で行う
- 1PR=1トピック。変更理由と範囲を明記する

## Requirements（要件）
- このプロジェクトで「やらないこと」も明記する

## Knowledge（知見の棚）
- 失敗例：
  - 例）グローバルに依存パッケージを入れて壊した
  - 例）仮想環境を無視してコマンドを実行した
  - 例）テストなしでコードを変更した
- 再発防止ルール：
  - 例）仮想環境を必須とする
  - 例）変更前にテストを実行する
  - 例）危険操作前に確認を求める
```

## ルール改善サイクルの詳細プロセス

### ステップ1: 指示理解
- ユーザーからの指示を受け取る
- 例：「こういうことやってください」

### ステップ2: コード実行
- ファイルの中身を編集
- Webサーチ
- その他の作業実行

### ステップ3: エラー発生
**動画で具体的に言及された問題例**:
- パイソン実行時に仮想環境を使わない
- グローバルインストールを繰り返す
- Geminiが「私にはできないからお前がやってくれ」と返答

### ステップ4: フィードバック
**フィードバックの種類**:
1. **人間からのフィードバック**: 直接的な指示や修正
2. **AIの自己内省**: AI自身が問題を分析し解決策を提案
3. **経験的知見の蓄積**: 他の場面でも使える知識の抽出

### ステップ5: 解決策整理
- 頻発する問題とその解決策を整理
- 生産性向上のアイデアを発見

### ステップ6: ルール反映
- 一定のやり取りが終わった後に必ずルールに反映
- 次の指示を待つ

## 実践的な実装方法

### 1. ルール更新のタイミング
- **作業終了時**: 必ず学習内容をルールに反映
- **問題発生時**: 即座に失敗例と解決策を記録
- **定期的**: 週次または月次でルールの見直し

### 2. フィードバック収集の仕組み
```
作業完了後の確認事項：
1. 何か問題は発生しましたか？
2. 同じ問題が繰り返されましたか？
3. 新しい解決策を発見しましたか？
4. ルールに追加すべき知見はありますか？
```

### 3. ルール更新の具体的手順
1. 問題の詳細を記録
2. 解決策を明確化
3. 再発防止策を策定
4. Knowledge（知見の棚）に追加
5. 関連する他のルールも更新

## ローカルファイル管理の重要性

### 動画で強調されたポイント
> 「独自コンテクストをローカルファイルとして自由に管理できるからこそできるわけなんですけど」

### 具体的な利点
- **複数ファイルに分けて管理可能**
- **自分なりの構造を定義可能**
- **システムプロンプトに書き出す必要がない**
- **Web系サービスでは実現困難**

### Web系サービスとの比較
- **Web系サービス**: システムプロンプトへの書き出しが必要
- **ローカルAIエディタ**: ファイルベースでの自由な管理

## よくある問題と解決策

### 問題1: 仮想環境を使わない
**症状**: Python実行時に仮想環境を無視
**解決策**: Environment（環境）ルールに「仮想環境を必須とする」を明記

### 問題2: 同じエラーの繰り返し
**症状**: 同じミスを何度も繰り返す
**解決策**: 失敗例をKnowledge（知見の棚）に記録し、再発防止ルールを追加

### 問題3: 危険操作の実行
**症状**: 削除や上書きを確認なしで実行
**解決策**: Environment（環境）ルールに「危険操作前は必ず確認」を明記

## 効果測定の指標

### 定量的指標
- **エラー発生率の減少**: 同じエラーの繰り返し発生回数
- **問題解決時間の短縮**: 初回発生から解決までの時間
- **ルール更新頻度**: 月次のルール追加・更新回数

### 定性的指標
- **AIエージェントの理解度向上**: 指示の正確な理解率
- **作業効率の向上**: 手戻りの減少
- **学習効果**: 新しい問題への対応能力

## 実装のベストプラクティス

### 1. ルールの粒度
- **細かすぎる**: 管理が困難
- **粗すぎる**: 効果が薄い
- **適切な粒度**: 具体的だが応用可能

### 2. 更新の頻度
- **毎回更新**: 管理負荷が高い
- **定期的更新**: 週次または月次が適切
- **問題発生時**: 即座に更新

### 3. ルールの優先度
- **Environment（環境）**: 最優先
- **Implementation（実装）**: 高優先度
- **Knowledge（知見の棚）**: 継続的更新

## 注意点と制限事項

### 動画で言及された制限
- スピーカーの経験は研究目的でのコーディングが中心
- プロダクト開発の経験は限定的
- 1ヶ月程度の実践経験に基づく手法

### 検証が必要な点
- 大規模プロジェクトでの適用性
- チーム開発での効果
- 長期的な効果の検証

## まとめ

ルール改善のサイクルは、AI駆動開発において最も重要な要素の一つです。完璧を目指すのではなく、使用しながら継続的に改善する仕組みを構築することで、AIエージェントとの協働がより効率的になります。

**重要なポイント**:
1. **ローカルファイル管理の活用**: 最大の強み
2. **継続的なフィードバック**: 改善の源泉
3. **経験的知見の蓄積**: 長期的な価値
4. **段階的な改善**: 完璧を目指さない

このサイクルを継続的に回すことで、AIエージェントは使用するたびに改善され、最終的に「最強」の状態に近づいていきます。
