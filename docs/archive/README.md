# 開発資料アーカイブ

## 📋 概要

ハッカソン開発で使用する既存の参考資料（30程度のtxt/mdファイル）を暫定的に格納する場所です。

## 📁 ディレクトリ構造

```
docs/archive/
├── README.md                    # このファイル
├── archive-instructions.md      # 格納手順書
├── raw-files/                   # 既存ファイルの一時格納場所
│   ├── txt/                     # .txtファイル
│   ├── md/                      # .mdファイル
│   └── other/                   # その他のファイル形式
└── categorized/                 # 構造化後の格納場所
    ├── requirements/            # 要件定義関連
    ├── technical/               # 技術資料
    ├── templates/               # テンプレート
    ├── references/              # 参考資料
    └── insights/                # 知見・ノウハウ
```

## 🔄 格納・構造化プロセス

### Phase 1: 一時格納
1. 既存の30程度のファイルを `raw-files/` に格納
2. ファイル形式別に分類（txt/, md/, other/）

### Phase 2: 構造化・格納
1. ファイル内容の分析・分類
2. `categorized/` 内の適切なディレクトリに移動
3. ファイル名の標準化
4. 内容の整理・統合

### Phase 3: 開発資料として活用
1. 構造化された資料を開発プロセスで活用
2. 必要に応じて `docs/requirements/`, `docs/references/` 等に移動
3. 継続的な更新・改善

## 📊 ファイル分類基準

### 要件定義関連
- プロジェクト要件
- 機能仕様
- 非機能要件
- 受け入れ条件

### 技術資料
- アーキテクチャ設計
- 技術選定資料
- 実装ガイド
- パフォーマンス要件

### テンプレート
- 開発計画テンプレート
- ドキュメントテンプレート
- コードテンプレート

### 参考資料
- 技術文書
- ベストプラクティス
- 事例研究
- 外部リンク

### 知見・ノウハウ
- 開発経験
- 失敗例・解決策
- 効率化手法
- 学習メモ

## 🚀 次のステップ

1. **一時格納**: 既存ファイルを `raw-files/` に移動
2. **内容分析**: 各ファイルの内容確認・分類
3. **構造化**: 適切なディレクトリに移動・整理
4. **開発活用**: ハッカソン開発プロセスでの活用

## 📝 注意事項

- ファイルの移動前に内容を確認
- 重複ファイルの統合を検討
- 機密情報の取り扱いに注意
- バージョン管理の徹底
