### **1. PDCAサイクル強制適用ルール**

#### **背景**
- ユーザー要請: 「今回のようなケースやタスクもIssueと解釈し、PDCAサイクルを回してください」
- Notion組織化タスク（Issue #48）からの学習

#### **確立されたルール**
```
すべてのタスク・課題をIssueとして記録し、PDCAサイクルを回す
```

**適用対象**:
- 技術的タスク（実装、設定、トラブルシューティング）
- 組織的タスク（整理、整理、ルール策定）
- 学習的タスク（新技術習得、ベストプラクティス導入）
- 運用タスク（ドキュメント更新、プロセス改善）

**判断基準**:
- ✅ 30分以上の作業
- ✅ 複数ステップが必要
- ✅ 学習要素を含む
- ✅ 将来の参考になる可能性

**実行手順**:
```
P (Plan): GitHub Issue作成、Notion記録（必要に応じて）
  ↓
D (Do): 実行、進捗をIssueにコメント
  ↓
C (Check): 成功要因特定、問題点分析、改善点抽出
  ↓
A (Act): ルール更新、テンプレート作成、知識共有、Issue Close
```

#### **実践例**

**Issue #48**: Notion組織化とAI向け説明書き追加
- P: Issue作成、計画明文化
- D: 組織化実行、説明書き追加
- C: 成功要因（段階的アプローチ）、改善点（早期説明書き）
- A: PDCAルール化、Issue Close

**Issue #49**: PDCAサイクル強制適用ルールの確立
- P: Issue #48の成功体験をルール化
- D: ルール策定、ドキュメント化
- C: メタ学習の重要性認識
- A: このIssue自体が実践例、Issue Close

#### **学習ポイント**
> 「PDCAサイクルを回すこと自体にもPDCAを適用する」

これにより、プロセス自体が継続的に改善される自己進化システムが確立される。

**ファイル**: `.cursor/rules/general.mdc`（60行追加）

### **2. Issue管理のベストプラクティス**

#### **発見された問題**
- 完了済みIssue #7-11が約1ヶ月Open状態で放置
- タイトル「【完了】」+ "completed"ラベルだがステータスがOpen
- PDCAサイクルが未完了（Issue Close未実施）

#### **根本原因**
1. 完了時にClose処理を失念
2. 定期的な棚卸しルール未確立
3. Issue管理の標準手順が不明確

#### **確立されたベストプラクティス**

**1. 完了時の即座Close**:
```
タスク完了
  ↓
成果物の確認
  ↓
即座にIssue Close（PDCAサイクル完全化）
```

**2. 月1回の棚卸し**:
```
月1回: Open Issues確認
  ├─ 完了済み → 即Close
  ├─ 継続中 → ステータス更新
  └─ 不要 → Close
```

**3. イベント終了時の一括整理**:
```
ハッカソン等終了
  ↓
関連Issueの一括整理
  ↓
完了コメント + Close
```

**4. タイトルとステータスの整合性**:
- 「【完了】」タイトル = 即Closed
- completedラベル = 即Closed

#### **実践結果（2025-10-12）**
- Issue #7-11: 完了済み5件を一括Close
- Issue #13-14: ハッカソン終了により2件Close
- Issue #15: テスト完了により1件Close
- **合計8件整理、Open Issues: 9件 → 2件に削減**

#### **学習ポイント**
- Issue管理の健全性維持が最重要
- 完了コメント追加でコンテキスト保持
- 定期的な整理でシンプルさを保つ

**関連Issue**: #7-15  
**ドキュメント**: `docs/2025-10-12-issue-cleanup-report.md`

### **3. 自己検証プロセスの確立**

#### **背景**
Issue #49の報告書作成時、ユーザーから検証要請:
> 「このまとめはチャットの内容であるはずです。矛盾や齟齬、ミスなどが無いか検証してください。」

#### **発見された誤り**
- Issueステータス: Open → 「完了」と誤記載
- Gitコミット数: 1件 → 「2件」と誤記載

#### **根本原因**
1. 推測に基づく記載
2. API/Git経由での事実確認不足
3. 自己検証プロセス未確立

#### **確立された検証プロセス**

```
報告書作成
  ↓
事実確認（GitHub API、Gitログ、チャット履歴）
  ↓
誤り発見
  ↓
修正実行（Issue Close、報告書修正）
  ↓
検証プロセス自体を文書化
  ↓
Git commit & push
```

#### **ハルシネーション回避手法**

**必須ステップ**:
1. **すべての数値をAPI/Git経由で取得**
   - GitHub API: Issue状態、作成日時、ラベル
   - Git: コミット履歴、変更行数
   
2. **推測ではなく事実に基づく記載**
   - 不確実な情報は明記
   - 確認実行してから記載
   
3. **誤りを発見した際は即座に修正**
   - Issue Close実行
   - 報告書修正
   - 検証プロセス文書化
   
4. **検証プロセス自体を標準化・文書化**
   - 再現可能な手順
   - 今後の参照用

#### **成功例（2025-10-12）**
- GitHub API経由でIssue #48, #49の状態確認
- `git log`でコミット履歴確認
- 誤りを発見（Open状態なのに「完了」と記載）
- 即座にClose実行
- 検証済み報告書を作成
- `docs/2025-10-12-pdca-system-establishment-report.md`

#### **学習ポイント**
> 報告書作成時は必ず事実確認。技術的合理性に基づく判断。自己検証が自己進化システムの基盤。

**再発防止ルール**:
1. **報告前の事実確認**: 数値はAPI/Gitで取得
2. **ユーザー要請の厳守**: 「検証してください」は必ず実施
3. **検証結果の記録**: 発見した誤りと修正を文書化

**関連Issue**: #49  
**ドキュメント**: `docs/2025-10-12-pdca-system-establishment-report.md`
