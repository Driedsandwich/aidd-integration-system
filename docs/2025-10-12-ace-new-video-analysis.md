## 📊 新動画の中核概念

### 1. コンテキストエンジニアリング（Context Engineering）

**定義**:
> 「エージェント中心の世界線での入力データ最適化」

**従来のプロンプトエンジニアリングとの違い**:
```
プロンプトエンジニアリング:
├─ タスクの指示
├─ 例示
└─ テキスト入力の最適化

コンテキストエンジニアリング:
├─ プロンプト
├─ システムプロンプト
├─ メッセージ履歴
├─ ドキュメント
├─ ツール情報
├─ ユーザー好みデータ（メモリ）
├─ ツール実行結果
└─ その他多様なデータソース
```

**重要性**:
- AI機能を持ったアプリケーション開発の中核
- AIを使って作業をこなす人にとっても重要
- 自己改善の根幹

---

### 2. コンテキストの崩壊（Context Collapse）

**既存手法の2つの問題**:

#### 問題1: Brevity Bias（簡潔性バイアス）
```
症状:
- 更新後のコンテキストが一般的で簡潔なものに偏る
- 重要な個別情報が消失

例:
Before: 「Gmailツールにはパラメーターがこれこれあって、
         過去に実行した時に必須パラメーターとして
         認識していなかったため失敗した」

After:  「Gmailツールを適切に使用する」
        ← 具体的な失敗情報が消失
```

#### 問題2: Context Collapse（コンテキスト崩壊）
```
症状:
- コンテキストが長くなるほど過剰に圧縮される
- 情報が急速に消失

論文での指摘:
繰り返しの更新により、コンテキストが大幅に縮小し
性能の急激な低下を引き起こす
（具体的な縮小幅は実験環境により異なる）
```

**根本原因**: 
> 「Monolithic Rewriting（一括書き換え）」
> LLMによるコンテキストの一括書き換えが、
> コンテキストをより短く情報量の少ない要約へと崩壊させる

---

### 3. Agentic Context Engineering (ACE)

**論文**: "Agentic Context Engineering: Evolving Contexts for Self-Improving Language Models"  
**著者**: Stanford University, SambaNova Systems, UC Berkeley

#### 核心的な解決策

##### A) 3つの役割分担

```
Generator（ジェネレーター）:
└─ 実際にタスクを解く担当

Reflector（リフレクター）:
└─ 内省・分析担当
   - 「ここに問題があった」
   - 「こうしたら良かった」
   - 「こうやったらうまくいった」

Curator（キュレーター）:
└─ コンテキスト更新担当
   - デルタ（差分）のみを出力
   - 新しいコンテキスト全体を出力しない
```

##### B) コンテキストの表現方法

**特徴**:
- 小項目の集まりとして定義
- セクション分けされた構造
- 意味的に重複する項目は削除（テキスト埋め込みで判定）

**実装例**（動画で示された構造）:
```markdown
## Strategies and Hard Rules (SHR)
- 明確なルール項目
- 使用回数カウント付き

## Coding Snippets
- 具体的なコード例
- ベストプラクティス

## Common Failure Modes
- 過去の失敗パターン
- 回避方法
```

**注**: セクション名は論文で規定されたものではなく、実装時の例示

##### C) Incremental Delta Updates（増分デルタ更新）

```
従来:
現在のコンテキスト全体 → LLM → 新しいコンテキスト全体
（問題: 一括書き換えで崩壊）

ACE:
現在のコンテキスト + 実行結果 → Curator → デルタのみ
（例: "Section X に項目Yを追加"）
```

**レスポンス形式の例**（動画・論文プロンプトより）:
```json
{
  "reasoning": "思考過程",
  "type": "add",
  "section": "セクション名",
  "content": "具体的な項目内容"
}
```

**注**: 上記は実装例であり、論文で厳密に規定された形式ではない

## 💡 実践的な応用（エンドユーザー向け）

動画から抽出された実践的なtips:

### Cursorルールの更新プロセス

**従来**:
```
「今やった作業内容をルールにして」
→ 一括書き換え → 崩壊のリスク
```

**ACE的アプローチ**:
```
Step 1: Reflect
「今やった作業を振り返って、
 どこに問題があったか、
 何がうまくいったか分析してください」

Step 2: Curate (差分生成)
「上記の分析結果をもとに、
 ルールのどこをどう書き換えたらいいか、
 変更差分を出してください」

Step 3: Apply
差分のみを適用
```

## 🎯 本ワークスペースへの影響評価

### 既存実装との整合性チェック

#### ✅ 整合している要素

1. **最小構成原則**
   - ACEも「incremental」「grow-and-refine」を重視
   - 完全一致

2. **RuleOps（ルール改善サイクル）**
   - ACEの自己改善ループと同一哲学
   - Generator→Reflector→Curator = 実行→振り返り→改善

3. **構造化されたコンテキスト**
   - 現在の`.cursor/rules/`のファイル分割
   - ACEの小項目集アプローチと類似

#### ⚠️ 新たに学べる要素

1. **3役割分担の明確化**
   ```
   現在: 暗黙的に分かれている
   ACE: 明示的に分離
   
   適用: ルール更新プロセスを3段階に明示化
   ```

2. **デルタ更新の重視**
   ```
   現在: ファイル全体の書き換えもある
   ACE: 差分のみの更新を推奨
   
   適用: search_replace の積極的使用
   ```

3. **コンテキスト崩壊の認識**
   ```
   現在: 意識していない
   ACE: 明確な問題として認識
   
   適用: ルールファイルの行数監視を正当化
   ```

## 📋 統合推奨事項（優先順位順）

### 🔴 Priority 1: 即座に統合すべき（今日中）

#### 1-1. knowledge.mdcに「コンテキスト崩壊」を記録

```markdown
## コンテキスト崩壊（Context Collapse）の認識

**問題**: 
ルールやコンテキストを一括書き換えすると、
詳細情報が失われ、簡潔すぎる要約になる

**原因**:
Monolithic Rewriting（一括書き換え）

**対策**:
1. 差分のみの更新（search_replaceの使用）
2. 3段階プロセス（実行→振り返り→差分生成）
3. 定期的な行数監視

**参考**: Agentic Context Engineering (Stanford, 2024)
```

#### 1-2. implementation.mdcに3段階プロセスを追加

```markdown
## コンテキスト（ルール）更新の3段階プロセス

### 従来の問題
一括書き換えは情報消失を招く

### ACE的アプローチ

1. **Reflect（振り返り）**
   - 何がうまくいったか
   - 何が問題だったか
   - どう改善すべきか

2. **Generate Delta（差分生成）**
   - どのファイルの
   - どの部分を
   - どう変更するか

3. **Apply（適用）**
   - search_replaceで差分のみ適用
   - ファイル全体の書き換えを避ける
```
