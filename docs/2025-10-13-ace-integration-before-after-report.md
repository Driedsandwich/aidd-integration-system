### 定性的変更

#### A) 新しい概念の追加

##### 1. コンテキスト崩壊（Context Collapse）

**Before**: 認識なし

**After**: 明確な問題として認識

```markdown
問題:
- Monolithic Rewriting（一括書き換え）
- Brevity Bias（簡潔性バイアス）
- 繰り返しの更新で情報が大幅に縮小

対策:
- 差分更新を優先
- ファイル全体の書き換けを避ける
```

**影響**: ルール更新の品質向上、情報消失の防止

---

##### 2. 3段階プロセス（Reflect → Delta → Apply）

**Before**: 暗黙的なプロセス
```
実行 → ルール更新
（1段階、直接的）
```

**After**: 明示的な3段階プロセス
```
Stage 1: Reflect（振り返り）
    ↓
Stage 2: Generate Delta（差分生成）
    ↓
Stage 3: Apply（適用）
```

**影響**: 
- ルール更新の品質向上
- 情報の保持（詳細が失われない）
- 構造化されたアプローチ

---

##### 3. 差分更新の原則化

**Before**: search_replaceを推奨（暗黙的）

**After**: 明確な原則として確立
```
✅ DO:
- search_replace優先
- 必要な部分のみ変更

❌ DON'T:
- writeでファイル全体書き換え
- 「簡潔にまとめる」指示
```

**影響**: 
- ツール選択の明確化
- コンテキスト崩壊の予防

---

#### B) 既存概念の強化

##### 1. 最小構成原則

**Before**: 
```
「最小限で開始 → 必要に応じて拡張」
- 一般的な原則
```

**After**: 
```
「最小限で開始 → 必要に応じて拡張」
+ コンテキスト崩壊回避の理論的裏付け
+ 差分更新の具体的方法
```

**影響**: 理論的基盤の強化

---

##### 2. RuleOps（ルール改善サイクル）

実行 → エラー → フィードバック → ルール反映

```

## 📈 実践レベルの変更

### ルール更新プロセス

#### Before

```
従来の方法:
1. タスク実行
2. 「今やった作業をルールにして」
3. ルール更新（一括書き換えの可能性）

リスク:
- 詳細情報の消失
- 簡潔すぎる要約
- コンテキスト崩壊
```

#### After

```
ACE的方法:
1. タスク実行
2. Reflect（振り返り）
   「何がうまくいった？問題は？」
3. Generate Delta（差分生成）
   「どこをどう変更すべき？」
4. Apply（適用）
   search_replaceで差分のみ

利点:
- 詳細情報の保持
- 具体的な失敗例の蓄積
- コンテキスト崩壊の回避
```

---

#### Before

```
暗黙的な推奨:
- search_replace推奨
- write使用も許容

リスク:
- writeで全体書き換え
- 意図しない情報消失
```

#### After

```
明確な原則:
✅ search_replace優先
   - 差分のみ更新
   - 情報保持

❌ write使用を最小限に
   - 全体書き換え回避
   - コンテキスト崩壊防止

根拠:
ACE論文のMonolithic Rewriting批判
